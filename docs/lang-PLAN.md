## Blog 项目业务扩展计划书（按投入产出优先级）

### Summary
- 目标：把当前“内容发布型博客”升级为“可增长、可转化、可复用”的内容平台。
- 核心原则：先做高 ROI 增长闭环，再做变现，再做平台化；架构演进不先于业务收益。
- 周期：12 周执行（P0/P1/P2 三阶段），每阶段均有明确上线目标、指标和退出条件。

### 分阶段路线（投入产出排序）
1. P0（第 1-4 周）：增长基础闭环（最高 ROI）
- 上线能力：
- 注册/登录闭环（管理员 + 普通用户）
- 评论、点赞、收藏（至少评论 + 点赞）
- 订阅入口（邮件订阅表单 + 留资）
- 内容分发基础（SEO 元信息、站点地图、RSS）
- 业务目标：
- 从“单向阅读”升级为“用户行为沉淀”
- 指标：
- 注册转化率、7 日回访率、文章互动率、订阅转化率
- 退出条件：
- 注册链路成功率 > 98%，互动功能可稳定运行 7 天

2. P1（第 5-8 周）：变现试点（中投入高产出）
- 上线能力：
- 会员/付费文章（先做最小版：单篇付费或专栏访问控制）
- 打赏入口（统一收款跳转）
- 广告位管理（首页/详情页固定位）
- 业务目标：
- 建立第一条收入通路，验证内容变现可行性
- 指标：
- 付费转化率、ARPU、广告点击率、打赏转化率
- 退出条件：
- 连续 2 周有稳定收入事件（非 0），并完成简单结算对账

3. P2（第 9-12 周）：效率与规模化（中 ROI）
- 上线能力：
- 推荐系统 v1（标签相似 + 热度推荐）
- 运营看板（渠道、内容、转化漏斗）
- 多角色后台（作者/编辑/管理员）+ 稿件状态流
- 业务目标：
- 提高内容分发效率与团队协作效率
- 指标：
- 人均发布效率、推荐点击率、阅读深度（PV/UV）
- 退出条件：
- 推荐点击率较基线提升，后台协作流程无阻塞

### 产品与接口规划（关键公共接口/数据模型）
- 用户域：
- `POST /api/v1/register`, `POST /api/v1/login`, `GET /api/v1/me`
- 模型：`user.role`（`ADMIN/USER/EDITOR` 预留）
- 互动域：
- `POST /api/v1/articles/{id}/comments`
- `POST /api/v1/articles/{id}/like`
- `POST /api/v1/articles/{id}/favorite`
- 变现域：
- `GET /api/v1/paywall/articles/{id}`
- `POST /api/v1/orders`
- `GET /api/v1/memberships/me`
- 分发与增长域：
- `GET /sitemap.xml`, `GET /rss.xml`
- `POST /api/v1/subscriptions`
- 推荐与分析域：
- `GET /api/v1/recommendations/articles/{id}`
- `GET /api/v1/admin/analytics/overview`

### 运营与实施机制
- 组织节奏：
- 每周一个小版本，双周一次指标复盘
- 每阶段设“业务 owner + 技术 owner + 数据 owner”
- 决策机制：
- 未达到阶段退出条件，不进入下一阶段大投入
- 指标明显不达标时，优先优化漏斗而非继续堆功能
- 发布策略：
- 新功能默认灰度（10% -> 50% -> 100%）

### 验收与测试计划（业务视角）
- 功能验收：
- 注册登录、互动、订阅、支付、推荐全链路可回放
- 数据验收：
- 埋点与后台看板一致；关键事件（注册、付费、订阅）可追踪
- 稳定性验收：
- 核心接口错误率、延迟、回滚预案可执行
- 业务验收：
- 每阶段核心指标达到预设阈值后再进入下一阶段

### 风险与应对
- 风险：
- 过早做复杂架构导致进度下降
- 功能上线但无数据闭环，无法判断收益
- 变现链路依赖外部支付/合规导致延期
- 应对：
- 保持“模块化单体 + 明确接口边界”
- 先埋点再扩功能；任何新能力必须定义指标
- 支付与会员先做最小可用方案，避免一次性大设计

### Assumptions
- 技术栈维持 Spring Boot + Vue + MySQL，短期不拆微服务。
- 先 B2C 内容增长与变现验证，再考虑平台化/SaaS 化。
- 当前团队资源按 1 后端 + 1 前端 + 1 产品运营核算。
